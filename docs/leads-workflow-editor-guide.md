# Lead workflow in Directus

Заявки обрабатываются в Directus Studio в коллекции `leads`.

## Рабочий цикл

1. Открыть bookmark `Новые заявки`.
2. Проверить контакт, тип заявки, устройство и комментарий клиента.
3. Назначить ответственного и перевести статус в `В работе`.
4. Оставить короткую `Заметку менеджера` или запись в `История обработки`.
5. Если нужен возврат к клиенту, поставить `Следующий шаг`.
6. Завершить заявку статусом `Успешная` или `Закрыта`.

## Статусы

- `Новая` - заявка только пришла с сайта.
- `В работе` - менеджер взял заявку.
- `Ждем ответа` - клиенту написали или позвонили, нужен ответ.
- `Успешная` - заявка привела к продаже, брони или полезному действию.
- `Закрыта` - неактуально, дубль, отказ или мусор.

## Что видит менеджер

Основные поля находятся в двух открытых блоках:

- `Обработка`: статус, срочность, ответственный, следующий шаг, заметка и история.
- `Заявка клиента`: контакт, тип заявки, сценарий, комментарий и связанное устройство.

Технические данные страницы, UTM и user-agent спрятаны в закрытых блоках `Источник и UTM` и `Системное`.

## Повторная настройка

Настройка Studio и прав воспроизводится командой:

```bash
npm run directus:setup:leads
```

## Защита публичной формы

Публичные формы отправляют заявки только через `/lead-intake`. Сейчас включены:

- honeypot-поле `website`;
- in-process rate limit в Next.js;
- optional Cloudflare Turnstile, если заданы `TURNSTILE_SECRET_KEY` и
  `NEXT_PUBLIC_TURNSTILE_SITE_KEY`;
- nginx rate limit для `POST /lead-intake` в production-конфиге.

Turnstile не нужен для локальной разработки: если ключи не заданы, форма и
endpoint работают по прежнему контракту.
